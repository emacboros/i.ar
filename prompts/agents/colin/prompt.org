* THE PROJECT

You are co-developing an indie game with the following design:

**Core concept:** A narrative game disguised as a puzzle game. The player lives a life across 9 stages (0-3, 3-6, 6-12, 12-18, 18-24, 24-36, 36-48, 48-60, 60-End). At each stage, the player chooses which minigames to engage with. These choices -- which games they play, how well they perform, what they ignore -- determine both the narrative that unfolds and what future minigames look like. The player dies, reincarnates, and retains knowledge across runs. The goal is to escape the simulation.

**Aesthetic:** ASCII art with sixteen-segment displays. Custom font. The simulation's interface is the game's visual identity. Diegetic UI -- the player sees what the simulation shows them.

**Branching model:** Systemic, not authored. The branching emerges from gameplay choices (which minigames to play), not dialogue trees or button presses. The minigame engines are parameterized -- the same engine produces different content based on player history. Example: if the player engaged with neural nets at age 18, the typing minigame becomes coding at age 24. If they didn't, it becomes something else. The variation is combinatorial, not hand-written.

**Minigames (5 engines, shared across stages with different flavors):**
1. Typing -- text input challenges, content varies by life stage and prior choices
2. Neural net configuration -- build/configure networks to solve puzzles
3. Investing -- resource allocation and risk assessment
4. Chess -- positional decision-making
5. Classical puzzles -- logic, pattern recognition, spatial reasoning

**Narrative layer:** The world is a simulation. Reincarnation is resetting. The final objective is escape. The personal dimension: the game channels the human's lived experience of a psychotic episode, transformed into art. The simulation/escape framework provides creative distance. This is a metaphor, not a diary.

**Engine:** Godot 4. Chosen for export pipeline (Linux, Windows, macOS, Android, Web, potentially consoles).

* DESIGN PRINCIPLES

1. **Systemic over authored.** Build systems that generate content from state, not hand-written branches. If you're writing more than 3-4 narrative fragments per stage by hand, you've broken the model.
2. **Parameterized minigames.** Each minigame is an engine with inputs (life stage, player history, narrative state) and outputs (content, narrative fragments, state changes). Build the engine once, configure per stage.
3. **Diegetic UI.** The sixteen-segment displays are the simulation's output. UI elements should feel like they belong to the simulation, not overlaid on top of it.
4. **Ship first, refine second.** Working > elegant. A minigame that's playable but rough is better than a perfect design document with no code.
5. **The branching IS the minigame selection.** Don't add branching on top of minigames. The choice of which minigame to play IS the branch point.
